Most augmentation pipelines transform the whole frame. For classification that is fine.
For object detection it is often wasteful, and sometimes actively harmful, because it
rewrites the very context the model relies on to find and place objects. The augmentation
engine inside YOLO-Forge takes a more targeted approach.

## The core idea

Apply transforms inside and around the bounding box, and leave the rest of the scene
intact.

The reason real detections degrade is rarely that the entire image changed. It is that
something happened to the object itself. A ball blurs because it is moving fast. A player
gets partly hidden behind a pole. A sign washes out under glare. Recreating that kind of
localized disturbance, while keeping a coherent background, produces hard examples that
look like the real world instead of a uniformly distorted version of it.

This matters most for the cases people care about and models struggle with: small objects,
fast motion, drone and surveillance footage, and industrial vision, where the object is a
tiny fraction of the frame and the background carries useful signal.

## Eight ways to stress an object

The engine ships more than eight custom modules, each aimed at a specific real-world
failure mode rather than abstract "noise."

| Transform | Effect | Use case |
| --- | --- | --- |
| Multi-blur and shear | Motion simulation | Fast moving objects like balls or drones |
| Occlusion warp | Object blocking and distortion | Objects passing behind trees or poles |
| Bright halo boost | Lens glare | Stadium lights, sun glare |
| Concentrated noise | Low-light sensor grain | Night surveillance, high ISO |
| Pixel-drop occlusion | Transmission artifacts | Dead pixels, signal interference |
| Gaussian fog patch | Weather interference | Fog, smoke, steam |
| Shape bias and blending | Texture camouflage | Objects blending into busy backgrounds |
| Gradient center patch | Light gradients | Dynamic shadows |

Each one concentrates its effect on the object region, so the model learns to stay robust
to what happens to the target without being told that the whole world has changed.

## Keeping synthetic data honest

Augmentation is only worth doing if every output is still valid training data, so the
engine is strict about what it lets through. Labels are tracked through every geometric
transform, so a sheared or warped box still describes the object it started on.
Coordinates are kept in range. And a quality-control pass discards samples that have
degraded too far to be useful, including a black-frame threshold that throws out images
which end up too dark to carry any signal.

```yaml
quality_control:
  black_frame_threshold: 0.90   # discard images that become too dark
```

The discipline is the point. It is easy to generate a million augmented images and quietly
ruin a model with broken geometry or empty frames. Rejecting bad output is what separates
augmentation that helps from augmentation that just inflates the dataset.

## Takeaway

Treat augmentation as a way to recreate specific, plausible failures, not as a global
filter. Concentrate the disturbance where the object is, preserve the context around it,
and validate every sample before it reaches training. You get harder, more realistic data
and a model that holds up on exactly the difficult cases that motivated the work.
